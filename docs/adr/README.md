# ADRs
This folder holds Architecture Decision Records.

## Purpose
ADRs capture why an important technical decision was made.

They are for:
- long-lived design choices
- major boundary decisions
- decisions with meaningful alternatives or tradeoffs

They are not for:
- status updates
- implementation logs
- every refactor

## Format
Use short numbered files:
- `0001-some-decision.md`
- `0002-another-decision.md`

Keep each ADR focused and short.

Suggested structure:
- Title
- Status
- Context
- Decision
- Consequences

## Rule
If a design decision is important enough that people are likely to re-argue it later, write an ADR.
